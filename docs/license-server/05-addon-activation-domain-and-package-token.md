# Phase 5 - Addon Activation, Domain Binding, and Package Token Flow

## Goal

Make paid Night Raven CMS addons, starting with Webshop, purchasable and
activatable in a secure flow. The Webshop activation screen should help the
admin buy a license key, bind it to the CMS installation, and receive/install
the private package without requiring a long-lived package token in the CMS.

## Current State

The CMS already has:

- `Webshop activation` UI
- `licenseKey` input
- `packageToken` input
- `requestWebshopLicenseActivation`
- `siteDomain`, `siteId`, and deployment platform sent to the license API
- entitlement storage in `webshop_addon_entitlements`

This phase changes the responsibility split.

## Target Flow

1. Admin opens `/dashboard/webshop`.
2. If not activated/expired, show:
   - `Buy webshop license key`
   - `Activate Webshop`
3. Buy button links to:

```text
https://www.nrcms.com/webshop?addon=webshop&domain=<signed-prefill>
```

4. The external NR CMS Webshop may use the domain as prefill, not as final
   proof.
5. Customer buys addon license.
6. Customer enters license key in CMS activation.
7. CMS calls license API:

```json
{
  "licenseKey": "...",
  "siteDomain": "example.com",
  "siteId": "stable-site-id",
  "deploymentPlatform": {...}
}
```

8. License server validates and binds license to site identity.
9. License server returns entitlement plus a short-lived package install token.
10. CMS stores entitlement, not package token.
11. Managed deploy pipeline uses token immediately. Self-hosted user gets clear
    instructions.

## Domain Binding

Domain is useful but not sufficient if it only comes from a query string.

Binding rules:

- First successful activation binds license to canonical domain + stable site ID.
- Re-activation from same site is allowed.
- Domain change requires admin support or a signed transfer flow.

Better proof options:

- Vercel OIDC for Vercel deployments.
- Self-hosted `.well-known/nr-cms-license-challenge` file challenge.
- Optional DNS TXT challenge for high-value licenses.

The MVP can bind on activation with site ID/domain and later add proof
challenges.

## Activation API

`POST /api/webshop/licenses/activate`

Request:

```json
{
  "licenseKey": "NRLS-...",
  "siteDomain": "https://example.com",
  "siteId": "stable-site-id",
  "deploymentPlatform": {
    "provider": "self_hosted",
    "projectId": "..."
  }
}
```

Response:

```json
{
  "entitlementToken": "signed-token",
  "expiresAt": "iso",
  "features": ["webshop"],
  "licenseKeyRef": "safe-reference",
  "packageName": "@nr-cms/webshop",
  "packageVersion": "0.10.0-private",
  "packageInstallToken": "short-lived-token",
  "packageInstallTokenExpiresAt": "iso"
}
```

Package token:

- generated by license server or package registry integration
- short-lived
- scoped to package name/version
- single-use if registry supports it
- never stored in CMS DB

## CMS UI Changes

Activation form:

- add `Buy webshop license key` button near title
- remove package token input from normal activation path
- optionally keep advanced/manual package token behind "Advanced self-hosted
  install" only if needed

Messages:

- expired license should show same buy/renew button
- activation success should distinguish:
  - package installed and ready
  - package install pending
  - self-hosted manual install required

## NR CMS Webshop Changes

The official `https://www.nrcms.com/webshop` store must:

- accept signed domain prefill
- carry requested domain through checkout metadata
- sell SKUs for 1, 2, 3, 6, 12, 24 months and lifetime
- call Night Raven License Server after payment
- pass addon product type ID, SKU, and domain/request metadata

Do not trust the incoming domain blindly. It is only a customer-provided desired
binding until activation verifies/binds it.

## Acceptance Criteria

- Activation page has Buy button.
- Buy link includes signed/encoded domain prefill.
- Activation no longer requires package token for normal managed flow.
- License server activation binds license to site identity.
- Package token is short-lived and not stored in CMS DB.
- Existing entitlement logic still supports ready/expired/install_pending.

## Tests

- buy link domain canonicalization
- activation request payload
- package token omitted from DB save
- expired license UI shows buy/renew path
- self-hosted fallback still works

Run:

```bash
npm run typecheck
npm run test
```

## Risks

- Package registry token generation depends on the chosen registry. Keep the
  license API response abstract until registry details are finalized.
- Domain transfer/support flow should be designed before selling expensive
  lifetime licenses.
